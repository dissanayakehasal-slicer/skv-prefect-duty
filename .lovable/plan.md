

## Problem

The `settings` table is empty. The admin-auth edge function expects two rows:
- `key = 'admin_password_hash'` → hashed version of `SKV#1902`
- `key = 'admin_password_changed'` → `'false'`

Without these rows, every login attempt returns "No admin password configured."

## Fix

Run a database migration to insert the two settings rows. The password hash will be generated by calling the edge function's `init` action, but since that action uses `UPDATE` (not `INSERT`), we need to first insert the rows, then call `init` to hash the password. 

**Simpler approach**: Insert the rows via migration, then trigger the `init` action from the edge function to populate the hash.

**Step 1 — Migration**: Insert placeholder rows into `settings`:
```sql
INSERT INTO settings (key, value) VALUES ('admin_password_hash', '') ON CONFLICT DO NOTHING;
INSERT INTO settings (key, value) VALUES ('admin_password_changed', 'false') ON CONFLICT DO NOTHING;
```

**Step 2 — Call init**: After migration, invoke the edge function with `{ action: 'init' }` to hash `SKV#1902` and store it. This can be done by curling the edge function or adding a one-time init call in the app.

Alternatively, we can add an **auto-init** to the edge function: if `verify` finds no row, automatically create and hash the default password, removing the need for a separate init step. This is the more robust approach.

### Recommended: Make the edge function self-initializing

In `supabase/functions/admin-auth/index.ts`, modify the `verify` action: if no `admin_password_hash` row exists, **insert** the default hashed password and `admin_password_changed = 'false'`, then proceed with verification. This ensures the system always works on first use without manual setup.
